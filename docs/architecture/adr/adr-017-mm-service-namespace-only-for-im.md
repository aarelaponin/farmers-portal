# ADR-017 — `mm_service` is namespace-only for the IM module

| | |
|---|---|
| Status | Accepted |
| Date | 2026-05-06 |
| Decider | Aare Laponin |
| Related | ADR-013 (metamodel reuse beyond RegBB); ADR-016 (IM uses kernel only); im-module SAD §4.4. |

## Context

In the RegBB framework, `mm_service` represents a citizen service offering. The framework reads `mm_service` rows to drive the single-window catalogue page (RegBB §6.1.6), the operator inbox grouping, the application form's wizard scaffold, and several other surfaces. The semantics matter — a `mm_service` is a thing citizens can register against.

ADR-016 establishes that IM does not use the RegBB framework. But IM's `mm_*` rows (determinants, actions, screens, fields, capabilities) still need to be **scoped together** so:

* Operators authoring IM rules see them grouped in the admin UI rather than mixed in with subsidy rules.
* Cross-module audit can answer "which rules are IM rules?" without parsing scope strings.
* Future modules following the same pattern have a precedent for namespace organisation.

`mm_service` already has a `code` column with no enforced semantic interpretation in the kernel — it's just a string. The framework gives it citizen-services meaning; the kernel doesn't.

## Decision

For the IM module, the `mm_service` row with `code='INPUTS_2025'` is **namespace-only**. It exists to satisfy the foreign-key contract from rows that scope themselves to it (`mm_determinant.serviceId='INPUTS_2025'`, `mm_action.serviceId='INPUTS_2025'`, etc.). It carries no citizen-services semantics:

* It does not appear on any single-window catalogue page (no citizen sees it).
* No `mm_registration` rows reference `INPUTS_2025` — IM has no sub-programmes-of-a-service.
* No application form is associated with it.
* No operator inbox surface displays it.
* The framework's `MetaModelDao.listRegistrationsForService("INPUTS_2025")` returns empty — by design.

The IM module's userview category ("Inputs Management") is hand-built (or generated by a future IM-specific publisher), independent of `mm_service`-driven generation.

## Alternatives considered

1. **Add a `mm_namespace` table separate from `mm_service`.** Rejected: multiplies the metamodel surface area for one bit of information (citizen-service vs. namespace-only). The `code` column on `mm_service` already carries the information; documentation suffices.

2. **Use a string prefix convention on rule codes — `IM_*` for IM, `SUBSIDY_*` for subsidy.** Rejected: weaker than a normalised `serviceId` foreign key. Rule codes aren't grouped at the schema level; cross-module reporting becomes a string-prefix scan.

3. **Inline-scope IM rules without referencing any `mm_service` row.** Rejected: rule-authoring UX inherits the `mm_service` selector by default; "no service" rows would be a special case throughout the admin surface.

## Consequences

**Positive:**

* IM rows group together via `serviceId='INPUTS_2025'` — cleanly queryable, cleanly displayable in the admin UI.
* No new schema introduced. No new authoring story for operators (they pick `INPUTS_2025` from the same `mm_service` dropdown they've already seen).
* The pattern generalises: future operational modules (e.g. crop monitoring, training delivery) declare their own `mm_service` row as namespace-only, populate `mm_determinant` and `mm_action` rows scoped to it, and inherit the rule + dispatch infrastructure without inheriting the citizen-services framing.

**Negative:**

* The `mm_service` table semantically mixes "citizen services" and "namespace bundles." A reader skimming the table can't immediately tell which is which without reading the description column. Mitigation: the module-of-origin convention (`SUBSIDY_*` codes for subsidy, `INPUTS_*` for IM) gives a quick visual signal; ADR-013 documents the dual-use contract.
* Future framework features that traverse `mm_service` rows must be aware that some are namespace-only. RegBB-specific traversals (catalogue page generation, applicability evaluation) currently key off `mm_registration` rows belonging to the service, so namespace-only services with zero registrations naturally drop out — no defensive code needed today. If a future feature traverses `mm_service` directly, it must filter to "services with at least one registration" or look at an explicit `kind` column added later.

**Trade-off named:** Schema simplicity vs. semantic precision. We chose schema simplicity (one `mm_service` table covering both senses) over a normalised but more complex metamodel (`mm_service` + `mm_namespace`). The naming is slightly misleading for IM, but the alternative — an additional table that exists solely to carry one bit ("citizen-service vs. namespace-only") — is overkill at current scale.
